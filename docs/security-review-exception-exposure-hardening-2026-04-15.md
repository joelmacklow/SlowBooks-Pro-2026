# Security Review — Exception Exposure Hardening (2026-04-15)

## Scope Reviewed
- `app/routes/companies.py`
- `app/routes/csv.py`
- `app/services/company_service.py`
- `tests/test_exception_exposure_hardening.py`

## Review Focus
- Whether company creation and CSV import routes still expose internal exception details to API clients
- Whether expected user-facing validation messages remain available where intentionally safe
- Whether the hardening changed normal success behavior or broadened trust boundaries

## Findings
1. **Company creation no longer leaks backend exception text**
   - The company service now separates internal `error` text from `public_error` and status code metadata.
   - The route returns only the controlled public message, preventing DSNs, credentials, or backend exception details from reaching clients.

2. **CSV import now converts decode and unexpected importer failures into safe HTTP errors**
   - Invalid encoding returns a specific safe 400 response (`CSV file must be UTF-8 encoded`).
   - Unexpected importer exceptions are mapped to a generic 500 response (`CSV import failed`) instead of surfacing internal exception strings.

3. **Safe validation/business-rule messages are preserved**
   - User-correctable company creation failures such as invalid database names or duplicate databases still return intentional public messages.
   - This keeps the API actionable without disclosing implementation details.

## Residual Risks
- Other routes in the codebase still contain explicit exception-string exposure patterns and should be reviewed separately.
- CSV importer per-row validation still returns row-specific messages generated by the importer; this slice only covers route-level exception exposure, not every downstream validation string.

## Conclusion
- No new CRITICAL/HIGH issues identified in this slice.
- Residual risk is **LOW to MEDIUM**, primarily because similar exception-exposure patterns may still exist elsewhere outside the patched routes.
