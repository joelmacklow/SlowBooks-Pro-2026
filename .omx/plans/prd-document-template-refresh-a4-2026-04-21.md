# PRD — Document template refresh from DOCX references (A4 / 1.5cm margins)

## Date
2026-04-21

## Objective
Refresh the printable/document PDF templates so invoices, quotes/estimates, credit notes, purchase orders, statements, and other generated documents visually align with the DOCX references in `../tmp/`, while standardising all document output to **A4** with **1.5cm margins**.

## Repo evidence
- Current document templates live in `app/templates/`:
  - `invoice_pdf.html`
  - `estimate_pdf.html` (quote analogue)
  - `credit_memo_pdf.html`
  - `purchase_order_pdf.html`
  - `statement_pdf.html`
  - `payroll_payslip_pdf.html`
  - `report_pdf.html`
  - legacy/stale variant: `invoice_pdf_v2.html`
- PDF generation is centralised in `app/services/pdf_service.py` and uses Jinja + WeasyPrint.
- Current page setup is inconsistent:
  - `invoice_pdf.html`, `estimate_pdf.html`, `credit_memo_pdf.html`, `purchase_order_pdf.html`, `statement_pdf.html` all use `@page { size: letter; ... }`
  - `payroll_payslip_pdf.html` already uses A4, but margins are not 1.5cm
  - `report_pdf.html` already uses `A4` and `1.5cm`
- Existing test coverage in `tests/test_pdf_service_formatting.py` already asserts A4/1.5cm for reports, but not for the other document families.

## Reference evidence from `../tmp`
Files provided:
- `Invoice.docx`
- `Quote.docx`
- `CreditNote.docx`
- `PurchaseOrder.docx`
- `Statement.docx`

Observed layout signals from the DOCX files:
- `Invoice.docx`, `Quote.docx`, `PurchaseOrder.docx`, and `Statement.docx` use **A4-sized pages** (`12240 x 15840 twips`), i.e. standard A4 portrait.
- These examples commonly use approx **1 inch left/right** and smaller top margins, but the requested implementation target is explicitly **1.5cm margins on all sides**.
- The invoice/statement family includes a **payment advice / remittance** style lower section in the reference examples.
- `PurchaseOrder.docx` includes a more operational **delivery details** section than the current HTML template.
- `Statement.docx` emphasizes aging / overdue/current/total summary blocks more strongly than the current statement template.
- `Quote.docx` is the visual reference for `estimate_pdf.html`.

## Problem
Current generated documents look like multiple independent templates from different eras:
- page size is mostly **letter**, not A4
- margins are inconsistent
- visual hierarchy differs document-to-document
- current HTML templates do not reflect several structural cues from the DOCX references (remittance/payment advice, stronger delivery sections, richer statement summary)
- style duplication across templates increases drift risk

## Scope
### In scope
- Refresh all **print/PDF document templates** generated by `pdf_service.py`
- Standardise **A4** page size and **1.5cm** margins for every document template in that family
- Align each document family to its corresponding DOCX reference where applicable:
  - Invoice -> `Invoice.docx`
  - Estimate/Quote -> `Quote.docx`
  - Credit Memo/Credit Note -> `CreditNote.docx`
  - Purchase Order -> `PurchaseOrder.docx`
  - Statement -> `Statement.docx`
- Bring payroll payslip into the same page setup standard even without a provided DOCX reference
- Decide what to do with `invoice_pdf_v2.html` (retire or intentionally preserve with explanation)

### Out of scope
- Editing DOCX files directly
- Replacing WeasyPrint
- Redesigning invoice email HTML unless implementation overlap makes that necessary
- Adding new document types beyond the existing generated PDFs

## Requirements
1. All generated document templates in the PDF service family must use `@page { size: A4; margin: 1.5cm; }` unless landscape is explicitly required for reports.
2. Invoice, quote/estimate, credit note, purchase order, and statement should share a visibly coherent design system:
   - typography scale
   - header composition
   - metadata tables/blocks
   - table styling
   - totals styling
   - footer treatment
3. Quote/estimate, invoice, and credit note should feel like one family rather than separate ad hoc layouts.
4. Statement should better reflect the provided reference emphasis on balance/aging/remittance information.
5. Purchase order should better reflect delivery/recipient context from the reference.
6. Existing dynamic fields, formatting filters, and HTML escaping behavior must remain intact.
7. Any shared CSS/partials introduced must reduce duplication without overcomplicating the template layer.

## Recommended implementation direction
### Option A — Minimal page-setup pass only
Change all templates to A4/1.5cm and leave structure mostly unchanged.

**Pros**
- smallest implementation risk

**Cons**
- fails the “see DOCX examples” intent
- visual inconsistency remains

### Option B — Recommended: document-system refresh
- create a shared document design language for PDF templates
- update each template to A4/1.5cm
- selectively incorporate structural cues from the references
- remove or retire stale variant templates where appropriate

**Pros**
- best alignment with user intent
- reduces long-term template drift
- gives consistent NZ-facing customer/vendor documents

**Cons**
- broader template diff
- requires stronger regression coverage

### Option C — Full componentised template refactor
- aggressively extract shared Jinja partials/macros for all document sections

**Pros**
- strongest reuse story

**Cons**
- higher risk of over-abstracting simple templates
- harder to review

## Recommendation
Choose **Option B**.

Use a light shared-template approach only where it clearly reduces duplication (for example shared page CSS tokens, company header/footer blocks, common totals styles), but avoid a heavy macro framework.

## Proposed phases
### Phase 1 — Page standardisation and inventory cleanup
- standardise all document templates to A4 / 1.5cm
- identify whether `invoice_pdf_v2.html` is dead/stale and retire it if safe
- add tests that lock A4/margin output across all templates

### Phase 2 — Family styling refresh
- unify invoice / quote / credit note styling
- modernise header, metadata, line table, totals, and footer consistency
- preserve existing data bindings and escaping

### Phase 3 — Reference-specific structural upgrades
- invoice / statement: payment advice / remittance-inspired section if supported by current data model
- statement: stronger balance summary / aging presentation
- purchase order: stronger delivery details layout
- quote: closer visual parity with invoice family while remaining quote-specific

### Phase 4 — Payroll/report alignment sweep
- bring payroll payslip to the same page/margin system and visual rhythm
- verify reports still intentionally allow landscape while keeping 1.5cm margins

## Acceptance criteria
1. All document templates generated by `pdf_service.py` use A4 with 1.5cm margins.
2. Invoice, estimate/quote, credit note, purchase order, statement, and payslip present a cohesive print design language.
3. The refreshed templates visibly move toward the DOCX examples rather than only changing page size.
4. Dynamic field rendering, NZ date/currency formatting, and HTML escaping continue to work.
5. Regression tests cover page setup and key reference-inspired content blocks.

## Risks and mitigations
- **Risk:** large template edits break escaping or formatting.  
  **Mitigation:** extend `test_pdf_service_formatting.py` instead of relying on visual guesswork only.
- **Risk:** reference DOCX sections imply data not currently available in models.  
  **Mitigation:** only implement reference sections that map cleanly to existing fields in this slice; document gaps explicitly.
- **Risk:** duplicated styles continue drifting.  
  **Mitigation:** extract only the most obvious shared CSS/partials.
- **Risk:** removing `invoice_pdf_v2.html` breaks an undiscovered path.  
  **Mitigation:** confirm usage before deletion; if uncertain, deprecate rather than delete in the first pass.
